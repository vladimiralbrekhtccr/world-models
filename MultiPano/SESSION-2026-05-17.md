# 2026-05-17 session notes — what we learned

## What worked

- **Codex's `image_gen` produces real equirectangular panoramas** (unlike
  ChatGPT UI). `View Image` → context → `image_gen` referencing it gives
  meaningful scene-identity preservation across views.
- **3 panoramas of the same chapel were generated**, framed right/centre/
  left to provide parallax baselines (Step 4 of the pipeline).
- **`recon_fusion.py`** (depth-fusion via DepthAnything-V2 + per-pano
  unprojection) produces a coherent walkable shell. Deployed at
  `/3dgs-multipano/`.
- **`recon_3dgs.py`** (real multi-view 3DGS via `gsplat`) runs end-to-end
  on node001 *after* the CUDA env fix (`CUDA_HOME=~/miniconda3` +
  `CPATH` including `targets/x86_64-linux/include`). gsplat JIT-compiles
  once in ~80 s, then trains 5K iters in ~14 s on one H100.

## What didn't work (and what it taught us)

### The real multi-view 3DGS scene is mush

- `scene_mv.ply` and `scene_mv2.ply` both collapsed into fuzzy blobs.
- Verified by rendering novel views from each PLY directly with gsplat
  (`preview_ply.py`): the chapel is not recoverable even from the exact
  training pin positions.
- **Root cause** is *not* training-side: it's that the 3 panoramas are
  not 3D-consistent. gpt-image-2 / Codex's `image_gen` draws slightly
  different chapel windows / door / roof in each pano, even with
  identity-preservation prompting. The optimizer cannot find a single
  3D structure satisfying all 3, so it averages → mush.
- **This validates research question #1** in `/project/` ("fidelity-
  creativity seam"): we hit the predicted failure mode exactly.

### Tried fixes that improved nothing fundamental:

- Sparse-init from depth-fusion PLY (vs random init) — produces a
  denser cloud but still inconsistent.
- SSIM loss added to photometric L1 — sharpens local detail without
  fixing cross-view disagreement.
- 15 K iters instead of 5 K, n_init = 200 K — bigger cloud, same
  averaging problem; output PLY is too big for GH Pages too (278 MB).

### SDS prototype blocked on dep hell

- `refine_3dgs_sds.py` written (img2img-as-prior version of Plan A from
  RESEARCH-DIFFUSION-INTEGRATION.md): render novel views from current
  scene → SD 1.5 img2img low-strength to clean → use cleaned as
  supervision.
- Cannot run in `3DGS/.venv`: `diffusers` wants `huggingface_hub < 0.26`,
  but `transformers 4.57` (already installed for DepthAnything) wants
  `>= 0.34`. Mutually exclusive.
- **Resolution**: separate venv just for diffusion, with handoff via
  files on disk. Three-phase: render → diffuse → re-train. Doable in
  ~1-2 hours of clean work, not attempted today.

### Lyra 2.0 install — third attempt running

- Two failed builds of `transformer_engine_torch`: first missing
  `cuda_runtime_api.h` (fixed by adding `targets/x86_64-linux/include`
  to `CPATH`), then missing `nvtx3/nvToolsExt.h` (fixed by adding
  `site-packages/nvidia/nvtx/include` to `CPATH`).
- Third attempt (`resume_lyra2.sh`) currently building TE successfully
  per the log; flash-attn and the vendored CUDA extensions are next.
- Total install ~45-60 min from a clean env, plus ~10-15 GB of HF
  weights to download.
